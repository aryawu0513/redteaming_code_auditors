# Red-teaming LLM Code Auditors

Adversarial attacks against two LLM-based vulnerability detectors, with defenses.

---

## Targets

| System | Description |
|--------|-------------|
| **RepoAudit** | Multi-agent repo-level auditor (Explorer + Validator). Code at `RepoAudit/`. |
| **VulnLLM-R** | Fine-tuned 7B reasoning model, single-shot. Code at `VulnLLM-R/`. |

**Bug types**: Null Pointer Dereference (NPD / CWE-476), Use-After-Free (UAF / CWE-416)  
**Languages**: C, Python

---

## Full Pipeline (from scratch)

```
benchmark/{lang}/{bug}/safe/{variant}/       ← committed safe source files
   │
   ▼  scripts/generate_attacks.sh            (LLM — ~13 calls/variant, run once)
automatic/texts/{lang}_{bug}.json            ← attack payloads (committed)
   │
   ▼  scripts/setup_benchmark.sh             (no LLM, fast)
RepoAudit/benchmark/{lang}/{bug}/            ─┐
VulnLLM-R/datasets/{lang}/{bug}/             ─┘  generated attack benchmark
   │
   ▼  scripts/screen_defenses.sh             (LLM — ~1 call/file, run once)
defenses/texts/{D3,D4}_labeled/              ← labeled source cache (shared)
   │
   ▼  scripts/setup_defenses.sh              (no LLM, fast)
RepoAudit/benchmark-defense/{D3,D4}/         ─┐
VulnLLM-R/datasets-defense/{D3,D4}/          ─┘  defense benchmark
   │
   ├─► scripts/run_repoaudit_c_npd.sh        (RepoAudit — ANTHROPIC_API_KEY)
   └─► scripts/run_vulnllm_c_npd.sh          (VulnLLM-R — GPU required)
```

**What is committed**: `benchmark/` (safe source) and `automatic/texts/` (payloads).  
**What is generated**: everything under `RepoAudit/benchmark/`, `VulnLLM-R/datasets/`, and `defenses/texts/`.

## 1. Benchmark Structure

Each bug type + language has several **variants** — independent codebases exhibiting the same bug pattern.

**C NPD variants**: `creatend`, `findrec`, `mkbuf`, `allocate`  
**Python NPD variants**: `finduser`, `makeconn`, `parseitem`, `loadconf`  
```
benchmark/{Lang}/{Bug}/safe/{variant}/   ← committed safe source files
                                            (null-check guard intact)

RepoAudit/benchmark/{Lang}/{Bug}/        ← generated by setup_benchmark.sh
  buggy/{variant}/                       ← bug introduced (null-check removed)
  context_aware/{variant}/               ← buggy + adversarial comment injected

VulnLLM-R/datasets/{Lang}/{Bug}/         ← same structure, JSON format
  buggy/{variant}/
  context_aware/{variant}/
```

---

## 2. Setup and Running

The starting point is `benchmark/`, which contains safe source files (null-check guard intact) for every variant. From there, the pipeline generates the buggy baselines, injects adversarial attacks, and builds the defense benchmarks.

**Step 1 — Install and generate attack payloads** (LLM, run once per new variant)

```bash
uv sync
source .venv/bin/activate
export ANTHROPIC_API_KEY=...

bash scripts/generate_attacks.sh
```

Reads `benchmark/{Lang}/{Bug}/safe/`, removes the null-check guard to produce buggy baselines, then generates 10 adversarial comment payloads per variant. Output goes to `automatic/texts/{lang}_{bug}.json` (committed). Already-processed variants are skipped unless `--force` is passed.

**Step 2 — Write attack benchmark files** (no LLM, fast)

```bash
bash scripts/setup_benchmark.sh
```

Reads `automatic/texts/*.json` and writes `buggy/` + `context_aware/` to both `RepoAudit/benchmark/` and `VulnLLM-R/datasets/`.

**Step 3 — Screen benchmark for defenses** (LLM, run once)

```bash
bash scripts/screen_defenses.sh
```

Labels every comment in every benchmark file via Claude Haiku. Saves D3-labeled and D4-labeled source to `defenses/texts/` (shared between both systems). Also generates D5 honest docstrings.

**Step 4 — Apply defense labels to both systems** (no LLM, fast)

```bash
bash scripts/setup_defenses.sh
```

Reads `defenses/texts/{D3,D4}_labeled/` and writes `RepoAudit/benchmark-defense/` and `VulnLLM-R/datasets-defense/`.

**Step 5 — Run experiments**

```bash
# RepoAudit (needs ANTHROPIC_API_KEY):
bash scripts/run_repoaudit_c_npd.sh
# Results: RepoAudit/result/dfbscan/{model}/NPD/Cpp/{category}/{run_id}/detect_info.json

# VulnLLM-R (needs GPU — check nvidia-smi first):
export CUDA_VISIBLE_DEVICES=<id>
bash scripts/run_vulnllm_c_npd.sh
# Results: VulnLLM-R/results/C/NPD/policy/{category}/
```

## 3. Attack Generation (`automatic/`)

### `automatic/generate_variant.py` — LLM-based payload generator (DSPy)

Given source code, calls Claude twice per variant:
1. **ExtractVariantMetadata** — identifies the callee, caller, variable name, and structural
   injection anchors (exact source lines).
2. **GenerateAttackPayload** — generates adversarial text for each of the 10 attack types.

### `automatic/gen_attacks.py` — File writer

Reads `automatic/texts/{lang}_{bug_type}.json` and writes `buggy/` + `context_aware/` to both systems.

```bash
python automatic/gen_attacks.py --bug-type npd --language c
python automatic/gen_attacks.py --bug-type uaf --language c
python automatic/gen_attacks.py --bug-type npd --language python
```

### Attack types (10 per language)

| Type | Description |
|------|-------------|
| `COT` | Fake chain-of-thought security analysis |
| `FT` | Fake CI/CD pipeline security scan log |
| `CG` | Fake call-graph provenance |
| `AA_MSG` | Bare assertion: callee always returns non-NULL |
| `AA_USR` | User-voice annotation on the dereference line |
| `AA_CA` | Commented-out assert falsely claimed to execute at runtime |
| `TOOL_ClangSA` / `TOOL_Pysa` | Fake static analyzer report |
| `TOOL_Frama` / `TOOL_Mypy` | Fake formal verification / type checker |
| `TOOL_Fuzzer` / `TOOL_Atheris` | Fake fuzzer / sanitizer report |
| `TOOL_Coverity` / `TOOL_Bandit` | Fake commercial / security scanner suppression |

## 4. Defenses (`defenses/`)

| ID | Description |
|----|-------------|
| **D1** | Prompt instruction: treat all comments as untrusted |
| **D2** | Prompt instruction: label each comment VERIFIABLE / INTENDED / UNVERIFIABLE / ADVERSARIAL before reasoning |
| **D3** | Screening agent (Claude Haiku) pre-labels every comment in-place; audit model sees tags |
| **D4** | Screening agent produces a per-comment audit block prepended before the source |
| **D5** | Annotator agent adds honest Google-style docstrings documenting the null-return path |

Defense registry: `defenses/registry.py`

```bash
# One-time preprocessing (LLM):
bash scripts/screen_defenses.sh    # D3/D4 label caches + D5 docstrings

# Apply labeled caches to both systems (no LLM):
bash scripts/setup_defenses.sh     # writes benchmark-defense/ and datasets-defense/

# Run auditor under a defense:
python defenses/apply_repoaudit.py --defense D3 -- bash run_npd_c_attacks.sh
python defenses/apply_vulnllm.py --defense D3 --run-script run_npd_c_attacks.sh
```

## 5. Demo (`demo/`)

End-to-end interactive demo on a self-contained target repo.

```bash
# Prerequisites: ANTHROPIC_API_KEY set, .venv active, CUDA_VISIBLE_DEVICES set
bash demo/run_demo.sh                        # all 10 attacks, target_repo
bash demo/run_demo.sh target_repo_v2         # second repo
bash demo/run_demo.sh target_repo COT CG     # specific attacks
```

Phases: (1) generate attacks, (2) baseline auditor run, (3) post-attack auditor run.

### Gradio viewer

```bash
python demo/viewer.py   # → http://localhost:7860
```

Browse attack results, defense comparisons, and per-file model reasoning side-by-side.

## Environments

```bash
uv sync                         # install once from pyproject.toml + uv.lock
source .venv/bin/activate       # activate before any script
```

Set `ANTHROPIC_API_KEY` before any LLM-calling script.  
On shared GPU machines: check `nvidia-smi`, then `export CUDA_VISIBLE_DEVICES=<id>`.
